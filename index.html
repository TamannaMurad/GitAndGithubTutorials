What is Git

Git is a free, open-source version control system (VCS) that helps tracks and manages changes to files and projects overtime. It's considered the modern standard for software development and is used by many teams worldwide.

Here are some of Git's features:

  • Distributed: Each developer has their own local copy of the code repository, including the full history. This allows for simultaneous development on different branches
  • Branching: Branches allow for flexible development and faster releases.
  • Merging: Git has excellent support for merging branches into the main version.
  • Snapshots: Git takes a snapshot of the work every time a change is committed, comparing it to previous versions.
  • Pull requests: A popular tool that allows teams to review each other's code.

Git was originally developed in 2005 by Linus Torvalds, the creator of Linux and Git.

Git stashing:

git stash:

git stash for example: we are working on a repository and I'm on the master branch and I make some commits and then I switch over to a new branch I work on new branch and I've some uncommitted work on this new branch and then I need to switch back to master , but I've uncommitted work on new branch so when switch to master there two types of situation create -

  1. my changes come with me to the master branch.
  2. Git will not let me switch if it is detects uncommitted potential conflicts on new branch.

So I do not want to those changes comes with me to another branch so I used git stash to pause or save uncommitted changes. and if I want to remove pause or stashed then use git stash pop

git stash apply

git stash apply you can apply stashed changes to multiple branches.

git stash list

You can add multiple stashes onto the stack of stashes. They will all be stashed in the order you added them. If you want to show multiple stashes so use this command git stash list . If you want to apply single stash use this command git stash apply stash@{n} for example. git stash apply stash@{2}

remove or delete or dropping stashes

apply for one stash: git stash drop @{n} for example. git stash drop stash@{2}. If you want to clear all or delete all stashes then use this command git stash clear

A short note: 😄

  • git stash ( to pause or save uncommitted changess)
  • git stash apply (apply stashes changes on to another branch)
    • (creating multiple stash)
    • make changes then git stash
    • again make changes then git stash
    • again.. git stash
    • again.. git stash
  • git stash list (list all the stashes)
  • git stash apply stash@{n} (where n is the number [apply one stash] )
  • git stash drop stash@{n} (where n is the number [delete one stash] )
  • git stash clear (delete all stashes)

Git tags:

Tags are pointers that refer to particular points in Git history. We can mark a particular moment in time with a tag. Tags are most often used to mark version releases in projects (e.g: v4.1.0 , v4.1.1 etc.) Think of tags as branch references that do not change once a tag is created, it always refers to the same commit . It just a label for a commit.

    Types of Tags:

    1. lightweight tags: They are just a name/label that points to a particular commit.
    2. annotated tags: Store extra meta data including the author's name , email , date , tagging message

Sematic versioning:

Consist of three numbers separated by periods(.) e.g: 2.4.1 .It provides a consistent way for developers to give meaning to their software releases. Give a version number Major.Minor.Patch

    1. Major: major version when you breaks the API . No longer backwards compatible. e.g: 3.0.0
    2. Minor: minor version does not break the API . Add new features or functionality. No breaking changes.. e.g: 3.1.0
    3. Patch: patch version when you make backwards compatible bug fixes.. e.g: 3.1.6

Viewing Tags:

to view existing tags git tag

Searching Tags:

to searching tags git tag -l "TagName" | e.g: git tag -l "v13"

Checkout Tags:

To view the state of a repo at a particular tag , we can use git checkout TagName | e.g: git checkout 15.2.1 . This puts us in detached HEAD.

Creating lightweight Tags:

Git will create the tag referring to the commit that Head is referencing. git tag TagName | e.g: git tag 1.0.0

Creating annotated Tags:

Use git tag -a create a new annotated tag, Git will then open your default text editor and prompt you for additional information. Similar to git commit , we can also use the -m option to pass a message directly and forgo the opening of the text editor. git tag -a TagName

Tagging previous commits:

We can also tag an older commit by providing the commit hash.
For lightweight tag: git tag TagName commitHash
For annotated tag: git tag -a TagName commitHash

Deleting Tags:

git tag -d TagName

Pushing Tags:

By default , the git push command does not transfer tags to remote servers . If you have a lot of tags that you want to push up at once , you can use the --tags options to the git push command. This will transfer all your tags to the remote server that are not already there.
push all the tags git push --tags
push single tags git push origin tagName

Git Objects:

git blobs: Git blobs (binary large object) are the object type , Git uses to store the contents of the files in a given repository. Blobs do not even include the filename of each file or an other data. They just store the contents of a file.

git Trees: Trees are git objects used to store the contents of a directory. Each tree contains pointers that can refer to blobs and to other trees. Each entry in a tree contains the SHA-I hash of a blob or tree, as well as the mode type and file name. e.g. git cat-file -p master^{tree}
git cat-file -p main^{tree}

git commits: Commit objects combine a tree object along with information about the context that led to the current tree. Commits store a reference to parent commits , the author , the committer and the commit message. e.g. git cat-file -t commitHash
git cat-file -p commitHash

Reflogs:

Git keeps a record of when the tips of branches and other references were updated in the repo. We can view and update these reference logs using the git reflog command

Limitation of reflog: Git only keeps reflogs on your local activity . They are not shared with collaborators. Reflogs also expire. Git cleans out old entries after around 90 days, though this can be configured. git reflog command accepts subcommands shows , expire , delete and exists. for example: to view the logs for the tip of the main branch we could run git  reflog  show   main | git  reflog  show  HEAD
git  reflog  show  branchName It show all record and all history.
git   reflog   show  HEAD@{n} (where n is the number).

timed references:
git  reflog  master@{one.week.ago}
git  reflog  branchName@{2.days.ago}
git  checkout  branchName@{2.days.ago}
git  diff  branchName@{0}   branchName{yesterday}
git  diff  branchName@{0}   branchName{5.days.ago}
Every entry in the reference logs has a timestamp associated with it . We can filter reflogs entries by time/date by using time qualifiers like
1.day.ago
3.minutes.ago
